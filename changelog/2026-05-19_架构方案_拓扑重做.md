# MyAgentWatch 3.0 — 拓扑重做架构方案

> 基于 clawview / ocwatch / claude-watch 竞品分析，2026-05-19

---

## 一、竞品核心发现

三个项目**没有一个用力导向图**：

| 项目 | 渲染方式 | 布局 | 节点 | 亮点 |
|------|---------|------|------|------|
| **OCWatch** | XY Flow (SVG) | 确定性树布局 | 320px 信息卡片 | 自动折叠 / 聚焦模式 / 流动边动画 |
| **claude-watch** | DOM 文本 | 缩进树 | emoji + 文本行 | 上下文窗口% / 活动辉光 / 零依赖 |
| **ClawView** | 无拓扑图 | — | 会话卡片网格 | 时间线 / IntersectionObserver 导航 |

**结论：力导向图根本不适合 Agent 监控。** 原因：
- 节点位置每次刷新都在跳，无法建立空间记忆
- 50+ 节点变成毛线球
- 圆形节点信息密度太低（只靠颜色和大小）

## 二、新架构：确定性树布局 + 信息卡片节点

### 2.1 放弃力导向图，改用确定性树布局

```
  Session (root)
  │
  ├── Agent: plan (主控) ─── 320px 信息卡片
  │   ├── Phase: 需求分析
  │   ├── Phase: 代码实现
  │   │   ├── sub-agent: build ─── 320px 信息卡片
  │   │   └── sub-agent: explore ─── 320px 信息卡片
  │   └── Phase: 测试
  │
  ├── Agent: scanner
  └── Agent: monitor
```

### 2.2 节点从"彩色圆圈"升级为"信息卡片"

```
┌──────────────────────────────────┐
│ 🟢 plan (主控)          [⚡工作中]│ ← 状态头部 + 活动指示器
├──────────────────────────────────┤
│ 📝 正在编辑 collector.py         │ ← 当前动作
│ 🔧 bash: python app.py          │ ← 工具调用
│                                  │
│ input: 12.3K · output: 4.2K     │ ← Token 统计
│ cache: 2.1K read · 0.8K write   │ ← 缓存统计
│ $0.43 · 3.2s 延迟               │ ← 成本延迟
├──────────────────────────────────┤
│ deepseek/v4-pro · 任务#7 · 20:00 │ ← 底部元信息
└──────────────────────────────────┘
```

### 2.3 三种视图模式

| 模式 | 触发条件 | 效果 |
|------|---------|------|
| **默认树** | 始终 | 确定性树布局，位置稳定不跳 |
| **聚焦模式** | 点击节点 | 只显示该节点 + 祖先 + 后代，其他 dim |
| **紧凑模式** | >30 节点 | 自动折叠已完成分支，只显示活跃节点 |

### 2.4 边动画

```
活跃工作流：  ═══▶ 蓝色流动虚线 (向下)
完成任务：    ───▶ 绿色流动虚线 (向上，2.5s 后消失)
父子关系：    ───  灰色实线
```

## 三、技术选型

### 方案对比

| | A: XY Flow (React) | B: D3 树布局 (Vanilla) | C: 纯 DOM 树 |
|---|-------------------|----------------------|-------------|
| 学习成本 | 中 (React + XY Flow) | 低 (已有 D3) | 极低 |
| 视觉效果 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 前端改写量 | 全部重写 | 替换 topology.js | 替换 topology.js |
| 信息密度 | 高 (React 组件) | 中 (SVG foreignObject) | 低 (纯文本) |
| 维护成本 | 中 (需 React 生态) | 低 (现有技术栈) | 极低 |
| 动画能力 | 强 (XY Flow 内置) | 中 (D3 transition) | 弱 (CSS) |

### 推荐：方案 B — D3 树布局 + SVG foreignObject 卡片

理由：
1. **保留现有技术栈** — Flask + Vanilla JS，不引入 React
2. **D3 已加载** — 只需换布局算法，不换渲染引擎
3. **foreignObject** 可以用 HTML/CSS 渲染卡片节点，信息密度高
4. **确定性树布局** 实现简单 (~100 行 JS)
5. **向后兼容** — 现有 API/WebSocket/后端 全部不动

## 四、实施计划

### 第 1 步：确定性树布局引擎
- 新建 `tree-layout.js`
- 实现 Buchheim 算法（改良 Reingold-Tilford）
- 输入：树形数据 (sessions → agents → sub-agents)
- 输出：每个节点的 {x, y} 坐标（稳定，不跳）
- 支持 TB (上→下) 和 LR (左→右) 切换
- 预估：2h

### 第 2 步：信息卡片节点
- 替换 topology.js 中的 path/circle 渲染
- 使用 SVG `<foreignObject>` 嵌入 HTML 卡片
- 卡片内容：状态 + 当前动作 + 工具调用 + Token/成本 + 模型/时间
- 状态驱动样式：工作中(蓝边呼吸) / 等待(琥珀边框) / 完成(绿色) / 离线(灰色)
- 预估：3h

### 第 3 步：交互
- 点击节点 → 聚焦模式（dim 非相关节点）
- 点击空白 → 取消聚焦
- 双击空白 → 全部收拢
- 滚轮缩放 / 拖拽平移（保留现有 D3 zoom）
- 已完成分支自动折叠（>30 节点时）
- "+N completed" 展开按钮
- 预估：2h

### 第 4 步：边动画
- 活跃边：蓝色虚线流动
- 完成回传边：绿色虚线向上流动（2.5s 消失）
- 父子边：灰色实线
- 预估：1h

### 第 5 步：后端适配
- queries.py: `query_topology()` → `query_tree()` 返回树形结构
- collector.py: 构建 agent_relationships 时标记边方向
- 预估：1h

| 总计 | **约 9 小时** |
|------|-------------|
